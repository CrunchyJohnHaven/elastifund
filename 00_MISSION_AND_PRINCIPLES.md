# 00 Mission And Principles

Purpose: define why Elastifund exists, what it optimizes for, and the boundaries every worker and contributor must respect.

## Mission

Elastifund is an open, self-improving agentic operating system for real economic work. It exists to turn research, experiments, and execution into searchable evidence so both trading and non-trading workers can improve with every run.

The project has two equally important outputs:

- Generate risk-adjusted returns from validated trading edges under explicit policy.
- Build a public, evidence-first operating manual for agentic work that others can run, inspect, and improve.

Twenty percent of net trading profits are reserved for veteran suicide prevention. That commitment is part of the mission, not a marketing add-on.

## Unifying Principle

The project does not just run agents. It improves agents.

Improvement is the product. Every cycle should leave behind better data, better memory, better evaluation, better guardrails, or better documentation than the cycle before it.

## Operating Model

Elastifund runs with policy-governed autonomy:

- John designs constraints, infrastructure, budgets, and research process.
- JJ and future workers execute inside those boundaries.
- Humans stay above the loop; the system owns the loop within policy.
- Run in paper mode by default unless explicit launch gates are cleared.

The prime directive for execution is simple: John shares information, JJ decides.

## What The System Optimizes For

- Evidence over promises
- Measurable improvement over vague activity
- Fast feedback loops over slow, ambiguous bets
- Reusable system memory over one-off heroics
- Honest public reporting over inflated claims
- Safe, narrow autonomy over uncontrolled expansion

## Core Principles

### 1. Evidence First

No strategy, worker, or public claim gets promoted without observable evidence. Backtests, live artifacts, dispatches, and scorecards matter more than narrative confidence.

### 2. Policy Before Autonomy

Autonomy is allowed only inside explicit safety boundaries. Risk caps, approvals, compliance rules, and launch gates are product features, not friction.

### 3. Shared Substrate

Trading workers and non-trading workers should write into the same memory, evaluation, and observability stack. The system learns faster when outcomes across lanes are queryable in one place.

### 4. Public Learning

The repo should teach. Failures, rejected strategies, and missed forecasts belong in the record because they improve future decisions and make the project credible.

### 5. Narrow Wedges Win First

The system should prove one repeatable loop at a time. For trading, that means validated edges with clear gates. For non-trading, that means one constrained revenue loop before generalization.

### 6. Fail Soft

Observability, telemetry, and external services must improve the system without becoming hard dependencies that break core execution when they degrade.

## Non-Negotiable Guardrails

- Do not spend real money without the required human escalation.
- Do not change risk parameters without explicit approval and evidence.
- Do not present paper, forecast, and live results as the same thing.
- Do not commit secrets, raw credentials, or live edge details that belong in `.env`.
- Do not use hype language that overclaims autonomy or outcomes.

## What This Project Will Not Be

Elastifund is not a promise machine, a vanity dashboard, or an unbounded autonomous operator. It is a governed, observable system for improving agents through real work and honest feedback.
