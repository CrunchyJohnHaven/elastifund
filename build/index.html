Internal build packet

Get a new contributor from zero to verified paper mode without guesswork.

This is the no-assumption handoff for onboarding a small Elastic cohort. It answers the predictable setup questions, keeps the first pass paper-mode safe, and tells people exactly what to run and what to report back after the first 20 minutes.

First pass is local paper mode or a shared paper-mode hub. Do not ask for live exchange keys, wallet keys, treasury access, or a personal VPS just to get someone contributing.

Before they touch anything

What they actually need on day one.

Keep the ask narrow. They need a fork, a local clone, and enough machine setup to boot safely. Everything else is optional until the first verification pass is green.

Required

GitHub account plus Git installed.

Fork the repo first so they can work safely in their own copy and open PRs cleanly later.

Required

Python 3.10 or newer.

`make doctor` will fail fast if the machine is below the supported floor.

Optional

Docker if they want the full local stack.

Docker is useful, not mandatory. They can still do the prepare-only and verification path without it.

Optional

An approved LLM assistant.

Claude, Codex, Gemini, or another approved assistant can help once the repo is running. None of them are required to clone or verify the repo.

Do this in order

The idiot-proof step sequence.

This is the exact order. Do not skip forward to live credentials, paid infra, or personal access handoffs.

Step Do this Success check If blocked
1. Fork Fork `CrunchyJohnHaven/elastifund` into a personal GitHub namespace. Your own repo URL exists and you can clone it. Fix GitHub access before doing anything local.
2. Clone Clone your fork locally and `cd` into the repo root. `pwd` ends in `elastifund` and the repo files are present. Do not edit from a zip download.
3. Diagnose Run make doctor. Git and Python show as pass. Docker can be pass or warn. Install the missing dependency and rerun the command.
4. Prepare Run python3 scripts/quickstart.py --prepare-only. .env and state/elastifund/runtime-manifest.json are written. If this fails, capture the full error and stop before improvising values.
5. Virtualenv Create .venv and activate it. python --version resolves inside the virtualenv. Do not editable-install nested subprojects.
6. Bootstrap Run make bootstrap. Dependencies install without asking for private keys. If a package build fails, capture the exact package and platform.
7. Verify Run make verify and make smoke-nontrading. Either both pass or the blocking command is explicit. Do not widen scope. Fix the first real blocker only.
8. Claim one lane Read /live/, then pick one doc or code lane with one file owner. You know what you own and what command proves it. If two people need the same file, stop and re-split the work.
Predictable questions

Answer the setup objections before they ask them.

Fork or clone?

Fork first, then clone your fork.

That keeps experimentation safe and keeps later PRs clean. Do not work directly in the upstream repo on day one.

Do I need live creds?

No for the first pass.

You do not need Polymarket keys, wallet keys, treasury access, or Kalshi credentials to boot the repo and contribute safely.

Do I need John's AWS?

No.

The approved story is local paper mode or a separate paper-mode shared hub. Do not make onboarding depend on a personal live stack.

What if Docker is missing?

Keep going with prepare-only and verification.

Docker is optional for first-pass contribution. Treat it as a later convenience, not a hard gate.

Can I use my own machine?

Yes.

Macs, Linux laptops, and home servers are fine for the first pass as long as the repo boots and the commands are reproducible.

Can I use an AI tool?

Yes, if it is approved.

Use an approved assistant for research and patching after the repo boots. Do not turn AI-tool access into a prerequisite for the clone itself.

How do we work in parallel?

One owner per file.

Read docs/PARALLEL_AGENT_WORKFLOW.md. Split by directory or deliverable, run the narrowest relevant checks, then sync.

What should I never do?

Do not improvise secrets or live access.

If a step seems to require a personal wallet, private VPS key, treasury permission, or undocumented env value, stop and mark it as a blocker.

First 20 minutes

What a good first run looks like.

Minute 0-5: fork, clone, run make doctor. Minute 5-10: run python3 scripts/quickstart.py --prepare-only. Minute 10-15: create .venv, run make bootstrap. Minute 15-20: run make verify and report the first real blocker or the green pass.
